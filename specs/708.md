# CEA-708 (CTA-708-E S-2023, with errata) — Implementation Digest

This document is a comprehensive digest of ANSI/CTA-708-E S-2023 ("Digital Television (DTV) Closed Captioning"), sufficient to guide a CEA-708 (DTVCC) encoder or decoder implementation without consulting the PDF for technical details. Section numbers in parentheses are cross-references back to the PDF.

CEA-708 is the digital-television closed captioning standard. It defines a layered protocol for delivering captions in a DTV bitstream and a window-based caption-rendering model that is far richer than CEA-608's character-grid model. CEA-708 also defines how a CEA-608 datastream may be carried alongside DTVCC data on the same transport channel (so DTV receivers with NTSC outputs can re-emit a valid CEA-608 line-21 signal).

For 608 details (Caption Mode, Text Mode, XDS, character sets, control codes), see specs/608.md.

---

## 1. Architecture

### 1.1 Layer model (§3)

CEA-708 is described as a 5-layer protocol stack, OSI-style. From bottom up:

| # | Layer            | Function                                                          |
|---|------------------|-------------------------------------------------------------------|
| 1 | Transport        | Extract DTVCC bytes from DTV video user-data; demux 608 from 708. |
| 2 | Packet           | Reassemble byte-aligned variable-length **Caption Channel Packets** (CCPs). Resync. |
| 3 | Service          | Demultiplex Service Blocks routed to up to 63 logical caption services. |
| 4 | Coding           | Parse syntactic elements from a service's byte stream (control codes, characters, multi-byte commands). |
| 5 | Interpretation   | Render and animate caption windows; manage pen/window state; sync. |

In addition to the 708 layers above, the Transport layer also carries a CEA-608 datastream that flows through layers 1 only (as a separate sub-channel) and is processed as defined in CEA-608. The DTVCC Transport Channel is the union of CEA-608 datastream + DTVCC Caption Channel.

### 1.2 Pre-allocated bandwidth (§4.1, §4.2)

Total DTVCC Transport Channel: nominally **9600 bps** (1200 bytes/s), divided by 1.001 for fractional frame rates (23.98/29.97/59.94 Hz). Of this, the CEA-608 datastream is allocated **960/1.001 bps** (120 byte-pairs/s — exactly the two byte-pairs/frame at 60 fields/s for both fields combined as 4 bytes per 30 frames). DTVCC captions get **at least 8640/1.001 bps**.

The 9600 bps is constant — present in every DTV frame regardless of whether captions are present. Padding is used to maintain it.

## 2. Transport layer (§4)

### 2.1 cc_data() syntax (§4.3, Table 2)

DTVCC bytes ride in `cc_data()` structures embedded in the video bitstream's user-data per ATSC A/53 (for ATSC) or in equivalent locations for other systems (SCTE 21, etc.). Bit-level layout:

```
cc_data() {
    reserved              : 1   = '1'
    process_cc_data_flag  : 1
    zero_bit              : 1   = '0'
    cc_count              : 5   (0..31)
    reserved              : 8   = 0xFF
    for (i = 0; i < cc_count; i++) {
        marker            : 1   = '1'
        reserved          : 4   = '1111'
        cc_valid          : 1
        cc_type           : 2
        cc_data_1         : 8
        cc_data_2         : 8
    }
}
```

Each iteration of the loop carries a single 16-bit "byte pair" plus its `cc_valid` flag and 2-bit `cc_type` tag. This makes each iteration 24 bits of payload (3 bytes) plus 5 bits of header bits per iteration, but only the byte pair counts against the 9600 bps DTVCC-payload allocation — the marker/reserved/cc_valid/cc_type bits are overhead outside that count.

`process_cc_data_flag` = 0 means discard the entire structure. `cc_count` is the number of byte pairs in this structure.

### 2.2 cc_type values (Table 3)

| cc_type | cc_valid | meaning                                                          |
|---------|----------|------------------------------------------------------------------|
| 00      | 0        | Field 1 — do not generate run-in clock + start bit (or DTVCC padding) |
| 00      | 1        | Field 1 CEA-608 byte pair                                        |
| 01      | 0        | Field 2 — same as 00 above but for field 2                       |
| 01      | 1        | Field 2 CEA-608 byte pair                                        |
| 10      | 0        | DTVCC padding                                                    |
| 10      | 1        | CCP **continuation** — both bytes are CCP data                   |
| 11      | 0        | DTVCC padding                                                    |
| 11      | 1        | CCP **start** — `cc_data_1` = CCP header byte; `cc_data_2` = first CCP data byte |

The contextual meaning of `cc_valid=0, cc_type=00 or 01` (no waveform / padding) depends on position in the structure: when it appears as a leading entry, it means "no 608 waveform on this field"; when it appears later, it is just padding.

### 2.3 Structure rules (§4.3.1)

1. CEA-608 byte pairs (cc_type 00 or 01) must appear at the **start** of `cc_data()`, in field-display order.
2. The number of CEA-608 byte pairs at the start is bounded by frame rate (§4.3.6, Table 4): typically 2 entries at 60p, 4 at 30i/30p, 4–6 at 24p, 4 at 30i with 3:2 pull-down, etc.
3. After all CEA-608 entries, the remainder is DTVCC: any number of CCP-start (`cc_type=11, cc_valid=1`) and CCP-continuation (`cc_type=10, cc_valid=1`) byte pairs, optionally separated/terminated by DTVCC padding (`cc_type=10 or 11, cc_valid=0`).
4. A CCP may span multiple `cc_data()` structures (continue in the next frame). The decoder reassembles CCPs across frames.
5. The end of a CCP is signaled either by reaching `packet_size_code` bytes total, or by encountering a new CCP-start byte pair before that count (which terminates the previous CCP early).
6. It is invalid for `cc_valid=1, cc_type=00 or 01` to appear *after* DTVCC bytes have started in the same `cc_data()`.

Padding byte values: `0x00, 0x00` is recommended for `cc_valid=0` entries.

### 2.4 CEA-608 datastream embedding (§4.3.2)

The decoder maintains two FIFO buffers: `NTSC_Line21_field1_data` and `NTSC_Line21_field2_data`. Byte pairs marked `cc_type=00, cc_valid=1` are pushed into the field-1 buffer; `cc_type=01, cc_valid=1` into the field-2 buffer. At each NTSC field-emission time, one byte pair is removed from the matching field's buffer and emitted on line 21.

* If the buffer is empty at field-emission time, output cc_data_1=0x80, cc_data_2=0x80 (two odd-parity nulls) plus the line-21 waveform.
* If `cc_valid=0` at the head of the FIFO, no line-21 waveform is generated for that field — *except* that if any field-2 caption waveform is generated, a field-1 waveform (clock run-in + start bits) must also be generated, per CEA-608 requirements.
* If a frame has no `cc_data()` at all, or `process_cc_data_flag=0`, the field buffers receive nothing for that frame.
* Encoders must not assume field buffers larger than 2 byte-pairs (4 bytes) per field. Encoders must deliver CEA-608 bytes at a substantially constant rate — exceeding 60/1.001 byte-pairs per second per field is forbidden, and downstream encoders and decoders rely on this.
* Field order on output is always interlaced field-1-then-field-2; for progressive sources, decoders may translate Top→F1 or Top→F2 at their option.

The full CEA-608 datastream (captions, text, XDS) should be preserved when transcoding NTSC to DTV. The 608 channel inside DTVCC is independent of the DTVCC caption services and is processed by an entirely separate decoder pipeline.

### 2.5 Latency (§4.4)

* End-to-end (encoder → display) ≤ 4 NTSC frames (≈ 4/30 s) for 60-Hz-based signals; ≤ 4 25-Hz frames for 50-Hz signals.
* Encoder delay alone ≤ 2 NTSC frames.
* Decoder delay alone ≤ 2 NTSC frames.
* Intermediate processors must not introduce systematic advance/delay unless compensating for a known constant elsewhere.
* For ATSC MPEG-2 streams, `vbv_delay` ≤ 45,000 (90 kHz ticks ≡ 0.5 s) keeps the picture's caption data within the 0.5 s buffer constraint.

### 2.6 Caption Service Metadata (§4.5)

Carried out-of-band (in PMT/EIT for ATSC). Defines:

* NUMBER OF SERVICES (1..16)
* For each service:
  * SERVICE NUMBER (1..31; 1..63 in 708 itself but only 1..16 fit in some transports' caption_service_descriptor)
  * TYPE OF SERVICE: 608 or 708. At most one 608 service may be signaled (the embedded 608 datastream).
  * For 708 services: LANGUAGE (ISO 639-2), DISPLAY ASPECT RATIO (4:3 or 16:9), EASY READER flag.

If metadata is missing, decoders should assume 16:9 aspect ratio. CEA-608 metadata absence has no effect on the encoding of 608 bytes onto NTSC outputs.

Decoders must offer the user the ability to select any of up to 16 CEA-708 services. CEA-608 channel selection support (CC1–CC4) is optional. The FCC requires service-descriptor acquisition from the EIT specifically (47 CFR §15.122).

### 2.7 Independent audio/caption control (§4.7)

Caption settings (selection, language) and audio settings (selection, language) must be independently controllable on the user side. Devices must offer a mode in which audio is unaffected by caption changes and vice versa.

## 3. Packet layer — Caption Channel Packets (§5)

### 3.1 CCP framing

A CCP is a variable-length byte sequence of total size 1..128 (including header) — the size is always even after the header. The first byte is the CCP header:

```
b7 b6  b5 b4 b3 b2 b1 b0
seq    packet_size_code
```

* `sequence_number` (2 bits) — rolls 0..3. Decoders use discontinuities to detect packet loss; on loss they perform a Reset for each existing service and resume at the next CCP start.
* `packet_size_code` (6 bits) — derives `packet_data_size`:
  * If `packet_size_code == 0` → `packet_data_size = 127`.
  * Else → `packet_data_size = packet_size_code * 2 - 1`.

`packet_data_size` is the number of bytes after the header (so `packet_data_size + 1` total bytes for the CCP). Each CCP's data area can hold a maximum of 127 bytes of payload.

### 3.2 CCP rules (§5.2)

* A CCP starts when a `cc_type=11, cc_valid=1` byte pair is seen at the cc_data() level. The first byte is the CCP header; the second is the first data byte.
* Subsequent bytes are CCP data (cc_type=10 byte pairs).
* A CCP ends when either `packet_data_size` data bytes have been received, or a new CCP header is encountered prematurely (which terminates the previous CCP — used at splices).
* If a syntactic element (a multi-byte command) is partially received when a CCP terminates, that element is discarded.
* Every syntactic element must be entirely contained within a single CCP. Service Block boundaries are *not* aligned with syntactic elements, but CCP boundaries *are*.
* Decoders should not reset on missing alignment between syntactic elements and CCPs unless they choose to, but unfinished elements may yield undefined/unexpected display.
* Discontinuity detection: decoders may use either or both of (a) sequence-number gaps modulo 4 (which misses multiples of 4 lost), or (b) detection that the parser still has a partial syntactic element at CCP start (treated as discontinuity since CCP boundaries should align with element boundaries).

## 4. Service layer (§6)

### 4.1 Services

* Up to 63 services (1–63). Service 0 is forbidden.
* Services 1–6 are "Standard": addressable via a 1-byte Service Block header.
* Services 7–63 are "Extended": addressable via a 2-byte Extended Service Block header.
* Service #1 = Primary Caption Service (verbatim primary-language captions).
* Service #2 = Secondary Language Service (translation or secondary-audio captions).
* Services 3–63 are at the discretion of the caption provider.

Only Services 1–6 must be decoded by minimum-implementation decoders (47 CFR §15.122 c)2). FCC mandates the directory be displayable for services 1..6.

### 4.2 Service Block syntax (Table 10, §6.2)

```
service_block() {
    service_number                  : 3   uimsbf
    block_size                      : 5   uimsbf       (0..31)
    if (service_number == 7 && block_size != 0) {
        null_fill                   : 2   = '00'
        extended_service_number     : 6   uimsbf       (7..63)
    }
    if (service_number != 0) {
        for (i = 0; i < block_size; i++) {
            block_data              : 8   bslbf
        }
    }
}
```

* Standard header (1 byte): high 3 bits = service_number 1..6, low 5 bits = block_size 1..31.
* Extended header (2 bytes): first byte = `0b111 bbbbb` (3 bits all 1 to mark extended; 5 bits block_size); second byte = `0b00 ssssss` (2 reserved bits + 6 bits extended service number 7..63).
* Null Block header = `0x00`. Indicates "no more service blocks in this CCP". When `service_number == 0`, `block_size` must also be 0.
* `block_size` is the number of byte-codes following the header in this block (not including the header).

### 4.3 Block placement in CCPs (§6.2.5)

* Service Blocks do **not** cross CCP boundaries. If a service has more data than fits, the encoder truncates and continues in the next CCP with a new Service Block header.
* Multiple Service Blocks for the same service may coexist in one CCP.
* If a CCP is not fully filled, the encoder should null-fill the rest of the buffer so that a Null Service Block Header naturally appears in unused space.
* A syntactic element (a multi-byte command) **may** cross Service Block boundaries within the same CCP; decoders must handle this. (Stream creators are encouraged to align elements within blocks for ease of generation, but decoders cannot assume this.)

## 5. Coding layer — code spaces (§7)

The 256-position code space is divided into four groups, each holding a "standard" set and an "extended" set. The standard set is reachable directly; the extended set requires the EXT1 prefix (0x10).

| Group | Range       | Standard set | Extended set | Comment                                       |
|-------|-------------|--------------|--------------|-----------------------------------------------|
| CL    | 0x00..0x1F  | C0           | C2           | Control codes                                 |
| GL    | 0x20..0x7F  | G0           | G2           | Graphic chars                                 |
| CR    | 0x80..0x9F  | C1           | C3           | Control codes                                 |
| GR    | 0xA0..0xFF  | G1           | G3           | Graphic chars                                 |

* `EXT1 = 0x10` (in C0). It causes the *next* byte to be interpreted in the extended set rather than the standard set: 0x00..0x1F → C2 (in CL); 0x20..0x7F → G2 (in GL); 0x80..0x9F → C3 (in CR); 0xA0..0xFF → G3 (in GR). EXT1 is active only for that one following byte.
* `P16 = 0x18` (in C0). Marks a 3-byte syntactic element where the next 2 bytes address a 16-bit-wide character set (reserved for future CJK).

### 5.1 Code Set C0 — miscellaneous control codes (Table 13, §7.1.4)

Defined codes (all in 0x00..0x1F):

| Hex  | Mnemonic | Bytes | Meaning                                           |
|------|----------|-------|---------------------------------------------------|
| 0x00 | NUL      | 1     | Null; no operation                                |
| 0x03 | ETX      | 1     | End of Text — terminates a caption text segment when no command follows |
| 0x08 | BS       | 1     | Backspace                                         |
| 0x0C | FF       | 1     | Form Feed — clear current window, move pen to (0,0) |
| 0x0D | CR       | 1     | Carriage Return — start of next row; rolls window if at edge; preserves pre-existing text on that row |
| 0x0E | HCR      | 1     | Horizontal Carriage Return — return to start of *current* row, erasing the row's contents |
| 0x10 | EXT1     | n+1   | Extension prefix; size depends on next byte (see below) |
| 0x18 | P16      | 3     | 16-bit character; next 2 bytes are the 16-bit code |

Length rules for *unused* codes in C0:
* 0x01..0x0F (except defined ones above): 1-byte syntactic element. Skip 1 byte.
* 0x11..0x17: 2-byte syntactic element. Skip 2 bytes.
* 0x18..0x1F (except defined ones above): 3-byte syntactic element. Skip 3 bytes.

So a decoder that doesn't recognize a C0 byte still knows how many bytes to skip.

CR in CEA-708 follows CEA-608's roll-up semantics if the new row would fall outside the visible window; if the next row is within the window and contains text, the cursor moves to the row's beginning but the pre-existing text is *not* erased (a key difference from CEA-608's CR). The exact "beginning of next row" depends on print direction — see Table 24 in the Interpretation layer §6.4.

HCR is similar to issuing N backspaces (to the start of the current row) followed by erasure of the column the cursor currently sits in — it erases the entire row including the rightmost column (which BS cannot reach). Conceptually like CEA-608's DER.

FF erases the current window and moves the pen to (0,0). Equivalent to ClearWindows(current) followed by SetPenLocation(0,0).

### 5.3 Code Set G0 — ASCII printable (Table 15, §7.1.6)

0x20..0x7F. Identical to ASCII *except* 0x7F = ♪ (music note) instead of DEL. Use 0x5F (`_`) as the substitution for unsupported G3 graphic symbols.

### 5.4 Code Set G1 — Latin-1 (Table 16, §7.1.7)

0xA0..0xFF. Identical to ISO-8859-1. 0xA0 = NBS (non-breaking space) — but since wordwrap is forced off in this version of the spec, NBS is treated identically to a regular space.

### 5.6 Code Set G3 — future expansion (Table 18, §7.1.9)

Currently contains only one defined character at 0xA0 (post-EXT1) — a closed-caption logo icon. Decoders should display 0x5F (underscore) in place of any unsupported G3 character.

### 5.7 Code Set C2 — extended control (§7.1.10, Tables 19–20)

Reserved for future control codes. Length rules (and byte-skip behavior for unimplemented decoders):

| Range (post-EXT1) | Length (incl. EXT1 + ExtCode + data) |
|-------------------|----------------------------------------|
| 0x00..0x07        | 2 bytes (no data)                      |
| 0x08..0x0F        | 3 bytes (1 data byte)                  |
| 0x10..0x17        | 4 bytes (2 data bytes)                 |
| 0x18..0x1F        | 5 bytes (3 data bytes)                 |

### 5.8 Code Set C3 — extended control 2 (§7.1.11, Tables 21–23)

Reserved for future control codes. Length rules:

| Range (post-EXT1) | Length and form                                |
|-------------------|------------------------------------------------|
| 0x80..0x87        | 6 bytes (4 data bytes)                         |
| 0x88..0x8F        | 7 bytes (5 data bytes)                         |
| 0x90..0x9F        | Variable: EXT1 + ExtCode + 1-byte header + N data bytes; N = (header & 0x3F) + 1; the high 2 bits of the header are a Type field (00=BOC, 01=COC, 10=EOC, 11=OSC); bit 5 of the header must be 0; bits 0..4 are length (0..27, but limited to 28 due to Service Block size). Multi-segment (BOC..EOC) commands for fonts/graphics download. |

The Type field (b7..b6 of the header byte) for variable-length commands lets a single command be split across segments: BOC, COC.., EOC. Single-segment commands use OSC (binary 11). Variable-length C3 command segments must not span Service Blocks (per §7.1.11.3.1), and total data length is limited to ≤ 28 bytes per command segment.

For unimplemented variable-length commands in 0x90..0x9F, decoders skip `(header_byte & 0x3F) + 1` bytes after the header.

## 6. Interpretation layer — windows, pens, sync (§8)

The Interpretation layer defines what the parsed commands and characters do — the on-screen rendering model.

### 6.1 Top-level model (§8.1)

* A **Caption Screen** is the entire display canvas (the safe-title area).
* A **Caption Window** is a rectangular subregion that holds caption text. Up to **8 windows per service**, IDs 0..7. Each can be visible or hidden.
* A **Caption Pen** holds the pen attributes (size, font, color, italic/underline/edge, offset, text tag) used when characters are drawn.
* **Caption Text** is written into the current window at the current pen location.
* **Caption Synchronization** controls when commands take effect: DLY/DLC/RST.

### 6.2 Screen coordinates (§8.2, §9.4)

The coordinate grid is mapped onto the SMPTE RP 218 "safe-title area" of the screen.

| Aspect | Grid (cols × rows)  | Min positioning resolution |
|--------|----------------------|----------------------------|
| 4:3    | 160 × 75             | 32 × 15                    |
| 16:9   | 210 × 75             | 42 × 15                    |
| Other  | 5×H × 75 where H = 32 × screen_width / 4 | H × 15 |

Origin (0,0) is the upper-left corner of the safe-title area. Coordinates are (horizontal, vertical) — note this order.

If a decoder uses the minimum (32×15 / 42×15) grid, it divides the encoded coordinates by 5 to get the cell. Caption providers must use the maximum (75×{160|210}) coordinate values; the decoder downsamples if its grid is coarser. A "super cell" in the minimum grid is 5×5 of the encoded grid.

If a window's resulting size exceeds the safe-title area, the decoder may modify pen attributes (italic→no, LARGE→STANDARD, etc.) to make it fit. As a last resort, a window may be disregarded entirely.

A caption authored only for 16:9 may be disregarded by a 4:3 decoder. Captions limited to 32 contiguous characters per row are 4:3-safe.

### 6.3 Caption windows (§8.4)

#### Anchor points (Table 13, §8.4.3)

Each window has an anchor point (0..8) that identifies one of nine reference positions on the window itself: corners, edge-midpoints, or center. Layout:

```
0 - top-left          1 - top-center       2 - top-right
3 - middle-left       4 - center           5 - middle-right
6 - bottom-left       7 - bottom-center    8 - bottom-right
```

The anchor is positioned at a screen coordinate (anchor_horizontal, anchor_vertical), and the window grows/shrinks from that anchor when fonts change. Solid edges of the window are bounded by the anchor; dashed (free) edges are where the window may grow or shrink.

#### Window size

* `row_count` (rc): number of rows, encoded as (count - 1). Range 0..14 (so 1..15 rows). Some legacy decoders limit to 12 rows.
* `column_count` (cc): number of columns, encoded as (count - 1). Range 0..31 for 4:3 (1..32 columns) and 0..41 for 16:9 (1..42 columns).
* "Virtual" rows/columns are character-cell-sized; physical screen size depends on the effective font size.

#### Row lock / column lock

In CEA-708-E, both are reserved/forced to YES. Earlier versions allowed unlocked windows that grew/shrank to accommodate font changes; this version mandates fixed row/column counts.

#### Wordwrap

In CEA-708-E, wordwrap is reserved/forced to NO (parameter must be set to 0 by encoders). Cursor only moves between rows on CR, FF, or SPL (SetPenLocation). Characters at the end of a row either replace the last character or are discarded.

#### Window priority (§8.4.2)

Each window has a priority 0..7 (0 = highest). When windows overlap, higher-priority is drawn on top. Equal-priority overlapping windows: decoder may relocate, hide one, or display in some order. Minimum decoder need not support overlapped windows — it may hide all but one.

#### Window visibility, display effects (§8.4.10)

Each window has a visible flag and a display effect:
* SNAP (default) — pop-on/pop-off.
* FADE — fade in/out at a configured speed.
* WIPE — wipe from a configured direction at a configured speed.

For WIPE: wiping-on uses `effect direction`; wiping-off uses the opposite direction. `effect speed` is in 0.5 s units, range 1..15 (so 0.5 s to 7.5 s).

Display effects are optional in minimum-implementation decoders, which may always SNAP.

#### Window colors / borders (§8.4.11)

* `fill_color` (R, G, B), each 0..3 — see §6.7.
* `fill_opacity`: SOLID, FLASH, TRANSLUCENT, TRANSPARENT.
* `border_type`: NONE, RAISED, DEPRESSED, UNIFORM, SHADOW_LEFT, SHADOW_RIGHT.
* `border_color` (R, G, B).

Minimum decoders must implement at least: solid black opaque (`fill_color=(0,0,0), fill_opacity=SOLID, border_type=NONE`) and transparent (`fill_opacity=TRANSPARENT, border_type=NONE`).

### 6.4 Print direction, scroll direction, cursor movement (§8.4.9, Table 24)

Eight print/scroll-direction combinations are valid. The pair determines how the cursor moves after writing a character (when justification is LEFT) and where CR sends the cursor:

| Print → Scroll              | Cursor advance (post-char)  | CR (carriage return) behavior            |
|-----------------------------|------------------------------|------------------------------------------|
| Left→Right + Top→Bottom     | column++                     | row--, column = 0                        |
| Left→Right + Bottom→Top     | column++                     | row++, column = 0                        |
| Right→Left + Top→Bottom     | column--                     | row--, column = max                      |
| Right→Left + Bottom→Top     | column--                     | row++, column = max                      |
| Top→Bottom + Left→Right     | row++                        | column--, row = 0                        |
| Top→Bottom + Right→Left     | row++                        | column++, row = 0                        |
| Bottom→Top + Left→Right     | row--                        | column--, row = max                      |
| Bottom→Top + Right→Left     | row--                        | column++, row = max                      |

NTSC-style roll-up captioning = print Left→Right + scroll Bottom→Top. A "ticker tape" effect = print Top→Bottom + scroll Right→Left.

Constraint: horizontal scroll is paired only with vertical print, and vice versa. Same-axis pairs are forbidden.

When CR causes the cursor to move outside the visible window, the window scrolls (in the scroll direction) and the cursor lands at the appropriate edge of the window.

Print direction or scroll direction changes via SWA on a window with existing text: recommended behavior is to clear the existing text and place the cursor at default position. Caption authors should issue ClearWindows or DeleteWindows before relying on direction changes.

Minimum decoder support: LEFT_TO_RIGHT printing, BOTTOM_TO_TOP scrolling.

### 6.5 Justification (§8.4.9.1)

Four justification modes: LEFT, RIGHT, CENTER, FULL. Names are in terms of left-to-right print direction; for other directions, "left" = start of row and "right" = end of row.

Behavior with respect to display timing:
* LEFT: each character is displayed as soon as received.
* RIGHT, CENTER, FULL: the decoder may display each character on receipt or wait until a "row completion indicator" arrives. A row completion indicator is: CR, ETX, or any command other than SPC, SPA, or an SPL within the same row.
* When a centered/right/full justified window receives a new character into a row that already contains text, the row should be cleared first, then the new content displayed.

If full justification is not implemented, treat full-justified as left.

### 6.6 Pen attributes (§8.5)

The pen state is a per-window record of the attributes and colors that apply to subsequent characters. Pen state changes only affect characters written *after* the change.

#### Pen size (§8.5.1)

Three sizes: SMALL, STANDARD, LARGE.

* STANDARD: max 1/15 of safe-title height tall; max 1/42 of safe-title width wide for 16:9 (1/32 for 4:3).
* LARGE: at most 42/32 × STANDARD in both dims (rounded to integer pels/lines).
* SMALL: at least 32/42 × STANDARD in both dims.
* All three pen sizes are mandatory (§9.12).

Recommended dimensions per common formats (Table 25):

| Aperture     | Aspect | Safe-title       | STANDARD W×H | LARGE W×H | SMALL W×H |
|--------------|--------|------------------|--------------|-----------|-----------|
| 1920×1080    | 16:9   | 1470×825         | 35×55        | 42×66     | 28×44     |
| 1280×720     | 16:9   | 1050×600         | 25×40        | 30×48     | 20×32     |
| 720×486      | 4:3    | 480×375          | 15×25        | 18×30     | 12×20     |

User pen-size override has highest priority. SPA-specified size has next priority.

#### Font styles (§8.5.3, §9.13)

Eight predefined styles, identified 0..7:

| ID | Style                               |
|----|-------------------------------------|
| 0  | Default (decoder choice)            |
| 1  | Monospaced with serifs (Courier)    |
| 2  | Proportional with serifs (Times)    |
| 3  | Monospaced sans serif (Helvetica)   |
| 4  | Proportional sans serif (Arial)     |
| 5  | Casual (Dom, Impress)               |
| 6  | Cursive (Coronet, Marigold)         |
| 7  | Small capitals (Engravers Gothic)   |

All 8 must be implemented; decoders may pick any typeface that fits the style. Users can override.

#### Character offset (§8.5.4)

SUBSCRIPT (0), NORMAL (1), SUPERSCRIPT (2). Optional; minimum-implementation decoders may ignore.

#### Pen styles

`italics` and `underline` are independent boolean flags. Italics and underline are required at minimum (§9.15).

#### Foreground and background

* `fg_color` (R,G,B), `fg_opacity`: SOLID, FLASH, TRANSLUCENT, TRANSPARENT.
* `bg_color` (R,G,B), `bg_opacity`: same four values.
* The character background is the small rectangular box around the character cell, separate from the window's fill.

#### Edge attributes (§8.5.8)

`edge_type`: NONE, RAISED, DEPRESSED, UNIFORM, LEFT_DROP_SHADOW, RIGHT_DROP_SHADOW (0..5). `edge_color` (R,G,B). Edge opacity matches `fg_opacity`.

#### Text tags (§8.5.9)

A 4-bit tag attached via SPA, indicating the *type* of caption text:

| Tag | Meaning                                              |
|-----|------------------------------------------------------|
| 0   | Dialog                                               |
| 1   | Source / speaker ID                                  |
| 2   | Electronically reproduced voice (phone, radio, PA)   |
| 3   | Dialog in language other than primary                |
| 4   | Voiceover (narrator NOT heard by characters)         |
| 5   | Audible Translation                                  |
| 6   | Subtitle Translation (translation into primary lang) |
| 7   | Voice quality description                            |
| 8   | Song lyrics                                          |
| 9   | Sound effect description                             |
| 10  | Musical score description                            |
| 11  | Expletive (decoder may opt out of displaying)        |
| 12..14 | undefined                                         |
| 15  | Text not to be displayed                             |

Decoders may apply automatic visual treatment (e.g., italics for "Source/speaker ID"). Decoder-defined attributes attached to a non-zero tag override the SPA's other attribute parameters; tag 0 (Dialog) does *not* override SPA. User overrides always trump decoder/tag/SPA values.

#### Pen attribute priority

User > Decoder text-tag mapping > SPA explicit values > predefined style.

### 6.7 Color representation (§8.8, §9.19)

Each color is a triple (R, G, B), each component encoded in 2 bits → 4 levels per channel → 64 colors total.

Minimum-implementation decoders must support at least Table 30's 8 colors:

| Color   | RGB     |
|---------|---------|
| Black   | (0,0,0) |
| White   | (2,2,2) |
| Red     | (2,0,0) |
| Green   | (0,2,0) |
| Blue    | (0,0,2) |
| Yellow  | (2,2,0) |
| Magenta | (2,0,2) |
| Cyan    | (0,2,2) |

Mapping rule for unsupported values when only the 8-color set is supported: 1→0, 2→2, 3→2 in each channel independently. Example: (1,2,3)→(0,2,2); (3,3,3)→(2,2,2); (1,1,1)→(0,0,0).

An optional 22-color expanded set (Table 31) adds dark/bright variants. Full 64-color RGB is also acceptable.

### 6.8 Service synchronization (§8.9)

* Each service has a **Service Input Buffer** of at least 128 bytes. Input bytes flow through this buffer to the command interpreter.
* `DLY n`: suspend interpretation of buffered data for `n × 0.1` seconds. New incoming data accumulates in the buffer; when delay expires (or the buffer fills, or DLC arrives, or RST arrives), interpretation resumes.
* `DLC`: cancel any active delay for this service. DLC must be detected *before* it enters the input buffer (otherwise it would be buffered and not recognized until after the delay expires).
* `RST`: reset the service. The reset clears the input buffer, removes all displayed windows for the service, deletes all defined windows, and clears all window/pen state.
* RST and DLC are pre-processed: a Reset / DelayCancel detector intercepts these single-byte commands at the ingress to each service's input buffer (Figure 21 in spec shows a hardware-style interpretation; software decoders implement it at parse time).
* Service is also reset on: channel change, Service Input Buffer overflow, or CCP sequence-number discontinuity.

### 6.9 SetPenLocation semantics (§8.7)

`SPL(row, column)` addresses a position in the *internal text buffer*, not on screen. (0,0) is the topmost-leftmost cell. Print direction does not affect SPL (which is row,column-addressed in the buffer regardless of how text flows on screen). Font size and proportional spacing do not affect SPL. Justification: when not LEFT and print direction is left↔right, the column parameter is ignored; when not LEFT and print direction is top↔bottom, the row parameter is ignored.

Don't author rows longer than the column count; don't issue SPL outside the defined window area — extended rows have no SPL representation.

## 7. DTVCC command set — byte-level specification

All commands are in the C1 code set (0x80..0x9F). Multi-byte commands store parameters in big-endian byte order. Bit numbering: b7 = MSB, b0 = LSB.

### 7.1 SetCurrentWindow — CW0..CW7 (§8.10.5.1)

* Opcodes: 0x80..0x87 (`0b10000 idx2 idx1 idx0`).
* Format: `CWn` (1 byte total).
* Effect: Sets the *current* window ID for this service. SWA, SPA, SPC, SPL, and caption text written subsequently are directed at this window.
* The window must already have been defined (by DefineWindow) for it to be addressable.

### 7.2 ClearWindows — CLW (§8.10.5.3)

* Opcode: 0x88. Length: 2 bytes (`0x88 windowmap`).
* `windowmap`: an 8-bit bitfield; bit `i` set → affect window `i`.
* Effect: For each window in the bitmap, remove all text and fill with the window's fill color. Reset pen location to (0,0).

### 7.3 DisplayWindows — DSW (§8.10.5.5)

* Opcode: 0x89. Length: 2. Format: `0x89 windowmap`.
* Effect: Make the bitmapped windows visible (using their configured display effect).

### 7.4 HideWindows — HDW (§8.10.5.6)

* Opcode: 0x8A. Length: 2. Format: `0x8A windowmap`.
* Effect: Hide (make invisible) the bitmapped windows. Definitions, content, and the current window ID are preserved.

### 7.5 ToggleWindows — TGW (§8.10.5.7)

* Opcode: 0x8B. Length: 2. Format: `0x8B windowmap`.
* Effect: Toggle visible/hidden state for each bitmapped window.

### 7.6 DeleteWindows — DLW (§8.10.5.4)

* Opcode: 0x8C. Length: 2. Format: `0x8C windowmap`.
* Effect: Delete (remove from display + erase definition for) each bitmapped window. If the current window is deleted, the current window ID becomes "uninitialized" — subsequent text won't go anywhere until SetCurrentWindow or DefineWindow is received.

### 7.7 Delay — DLY (§8.10.5.12)

* Opcode: 0x8D. Length: 2. Format: `0x8D t` where `t` ∈ 1..255.
* Effect: Suspend interpretation of this service's command buffer for `t / 10` seconds.

### 7.8 DelayCancel — DLC (§8.10.5.13)

* Opcode: 0x8E. Length: 1.
* Effect: Cancel any active delay for this service.

### 7.9 Reset — RST (§8.10.5.14)

* Opcode: 0x8F. Length: 1.
* Effect: Reset this service (see §6.8 of this digest).

### 7.10 SetPenAttributes — SPA (§8.10.5.9, errata)

* Opcode: 0x90. Length: 3. Format: `0x90 parm1 parm2`.
* parm1: `tt3 tt2 tt1 tt0 o1 o0 s1 s0`
* parm2: `i u et2 et1 et0 fs2 fs1 fs0`

Where:
* `s` (pen size, bits 1..0 of parm1): 0=SMALL, 1=STANDARD, 2=LARGE.
* `o` (offset, bits 3..2 of parm1): 0=SUBSCRIPT, 1=NORMAL, 2=SUPERSCRIPT.
* `tt` (text tag, bits 7..4 of parm1): 0..15 per §6.6.
* `fs` (font style, bits 2..0 of parm2): 0..7 per §6.6.
* `et` (edge type, bits 5..3 of parm2): 0=NONE, 1=RAISED, 2=DEPRESSED, 3=UNIFORM, 4=LEFT_DROP_SHADOW, 5=RIGHT_DROP_SHADOW.
* `u` (underline, bit 6 of parm2): 1=YES.
* `i` (italics, bit 7 of parm2): 1=YES.

### 7.11 SetPenColor — SPC (§8.10.5.10)

* Opcode: 0x91. Length: 4. Format: `0x91 parm1 parm2 parm3`.
* parm1: `fo1 fo0 fr1 fr0 fg1 fg0 fb1 fb0` — fg opacity, fg R, fg G, fg B (2 bits each).
* parm2: `bo1 bo0 br1 br0 bg1 bg0 bb1 bb0` — bg opacity + bg R/G/B.
* parm3: `0 0 er1 er0 eg1 eg0 eb1 eb0` — edge R/G/B (top 2 bits zero). Edge opacity = fg opacity.

Opacity values: 0=SOLID, 1=FLASH, 2=TRANSLUCENT, 3=TRANSPARENT.

### 7.12 SetPenLocation — SPL (§8.10.5.11)

* Opcode: 0x92. Length: 3. Format: `0x92 parm1 parm2`.
* parm1: `0 0 0 0 r3 r2 r1 r0` — row (0..14).
* parm2: `0 0 c5 c4 c3 c2 c1 c0` — column (0..31 for 4:3, 0..41 for 16:9).

### 7.13 SetWindowAttributes — SWA (§8.10.5.8)

* Opcode: 0x97. Length: 5. Format: `0x97 parm1 parm2 parm3 parm4`.
* parm1: `fo1 fo0 fr1 fr0 fg1 fg0 fb1 fb0` — fill opacity + fill R/G/B.
* parm2: `bt1 bt0 br1 br0 bg1 bg0 bb1 bb0` — border type LSBs + border R/G/B.
* parm3: `bt2 ww pd1 pd0 sd1 sd0 j1 j0` — border type MSB, wordwrap, print direction, scroll direction, justify.
* parm4: `es3 es2 es1 es0 ed1 ed0 de1 de0` — effect speed, effect direction, display effect.

Where:
* `j` (justify): 0=LEFT, 1=RIGHT, 2=CENTER, 3=FULL.
* `pd` (print direction): 0=LEFT_TO_RIGHT, 1=RIGHT_TO_LEFT, 2=TOP_TO_BOTTOM, 3=BOTTOM_TO_TOP.
* `sd` (scroll direction): same encoding as pd.
* `ww` (wordwrap): forced to 0 in this version.
* `de` (display effect): 0=SNAP, 1=FADE, 2=WIPE.
* `ed` (effect direction): same encoding as pd.
* `es` (effect speed): 1..15, in 0.5 s units.
* `bt` (border type, 3 bits assembled from parm3 b7 + parm2 b7..b6): 0=NONE, 1=RAISED, 2=DEPRESSED, 3=UNIFORM, 4=SHADOW_LEFT, 5=SHADOW_RIGHT.

### 7.14 DefineWindow — DF0..DF7 (§8.10.5.2)

* Opcodes: 0x98..0x9F (`0b10011 idx2 idx1 idx0`). Length: 7. Format: `0x9X parm1..parm6`.
* parm1: `0 0 v rl cl p2 p1 p0` — visible, row-lock, column-lock, priority.
* parm2: `rp av6 av5 av4 av3 av2 av1 av0` — relative-positioning flag + anchor vertical (0..74 absolute, 0..99 relative).
* parm3: `ah7..ah0` — anchor horizontal (0..209 for 16:9, 0..159 for 4:3, 0..99 if relative).
* parm4: `ap3 ap2 ap1 ap0 rc3 rc2 rc1 rc0` — anchor point (0..8) + row count (0..14, encoded as count-1).
* parm5: `0 0 cc5 cc4 cc3 cc2 cc1 cc0` — column count (0..31 for 4:3 / 0..41 for 16:9, encoded as count-1).
* parm6: `0 0 ws2 ws1 ws0 ps2 ps1 ps0` — window style ID (0..7) + pen style ID (0..7).

Effect:
* If the window doesn't exist, create it. Initialize all character positions to the fill color and pen location to (0,0).
* If the window exists, update parameters. Pen location and pen attributes are *not* affected by an update. If the new parameters are identical to the existing ones, the command must be treated as a no-op.
* Style IDs of 0:
  * On creation: defaults to style #1.
  * On update: leave existing attributes unchanged.
* Style IDs nonzero: load the predefined style preset (Tables 26–27, see below). Subsequent SWA / SPA can modify these.
* DefineWindow also makes the defined window the *current* window.
* Caption providers should periodically repeat DefineWindow / SWA / SPA so that channel-changers and tune-ins acquire complete state.

#### Predefined window styles (Table 26):

| ID | Justify | Print | Scroll  | WW  | Effect | Fill RGB | Fill opacity | Border | Comment                                       |
|----|---------|-------|---------|-----|--------|----------|--------------|--------|------------------------------------------------|
| 1  | LEFT    | L→R   | B→T     | NO  | SNAP   | (0,0,0)  | SOLID        | NONE   | NTSC pop-up                                    |

| 4  | LEFT    | L→R   | B→T     | YES | SNAP   | (0,0,0)  | SOLID        | NONE   | NTSC roll-up (legacy "wordwrap=YES"; this version forces NO) |
| 5  | LEFT    | L→R   | B→T     | YES | SNAP   | n/a      | TRANSPARENT  | NONE   | Roll-up w/o black background                   |

## 8. Caption text and termination (§8.6, §8.10.4)

* Displayable G0/G1/G2/G3 characters and the C0 codes NUL, BS, CR, HCR, FF, ETX, P16 are directed at the **current window**. Other (undefined) C0 codes are commands and may not be directed at the current window.
* Visibility is controlled solely by the window's visibility flag (DSW/HDW/TGW etc.); writing characters to a hidden window builds up text that will appear when the window is shown.
* A caption-text segment is terminated either by the start of a new caption command (any C1/C2/C3/extended code) or by an ETX (0x03) in C0. ETX is required when text trails to the end of a Service Block with no subsequent command — without it, decoders may wait for the next Service Block to see if more text follows (especially for centered/right/full-justified rows).

### 8.1 Cursor advance after writing a character (Table 24)

The cursor advances one cell in the print direction after each character (per §6.4). Once the cursor reaches the maximum column/row (depending on direction), additional characters either replace the last cell or are discarded — the spec leaves this implementation-defined.

## 9. Authoring patterns (§8.11)

Caption providers should treat each new caption as a self-contained "burst," so that channel-changers and tune-ins always acquire correctly. Recommended sequences:

### 9.1 Roll-up

```
DLW (delete all windows except the one being used)
DFx (define/redefine window x)
SWA (optional, customize)
SPL  (move pen to leftmost column of bottom row)
CR
[pen commands] [text]
```

CR can come immediately before the SPL instead. If SPL precedes CR, all rows scroll into the window as they're sent. If CR precedes SPL, the first row appears at the bottom without scrolling and subsequent rows scroll.

### 9.2 Paint-on

```
RST  (delete all windows)
DFx  (define the window)
SWA  (optional)
[pen commands] [text]
```

If other windows must persist, replace RST with DLW; insert CLW before pen commands to clear the current window.

### 9.3 Pop-on (uses two windows)

```
DLW  (delete all windows except the displayed one)
DFx  (define the hidden window)
SWA  (optional)
[pen commands] [text into hidden window]
CLW  (clear the displayed window — produces "blink")
TGW  (toggle: hide the displayed; show the previously hidden)
```

If a "blink" is undesired, omit CLW. A DLY may be needed between CLW and TGW to actually see the blink.

## 10. Decoder minimum requirements (§9, normative)

### 10.1 Services (§9.1, §9.2)

* Decode at least one service. Decode the Caption Service Directory metadata.
* Decode all service-block headers (Standard, Extended, Null), even when only services 1..6 will be processed.
* Allow user selection of any of the 4 CEA-608 caption channels (CC1..CC4) even without a service directory present.
* Don't interpret the 5-bit reserved field of `caption_service_descriptor()` — it may be used in future revisions.
* Allow service selection by number, even if no directory is present.
* May auto-switch from 708 to 608 when 708 captions are absent (algorithm out of scope).

### 10.2 Code space (§9.3)

* Support all of C0, G0, C1, G1.
* Support these G2 characters: TSP, NBTSP, █ (solid block), ™, Š, Œ, š, œ, Ÿ.
* For unsupported G2 chars, perform the substitutions in Table 28 (e.g. open quotes → straight quotes, ellipsis → underscore, fractions → percent, box drawings → dash/pipe).
* Skip unimplemented C2/C3/G3 syntactic elements per their length rules.
* Substitute unsupported G3 graphics with G0 underscore (0x5F).

### 10.3 Windows (§9.5..§9.10)

* Display ≥ 4 windows simultaneously. May ignore lower-priority/younger windows beyond that. Storage for ≥ 8 rows of caption (4 displayed + 4 buffered for next pop-on).
* May discard windows with `rc>3`, but not required.
* Don't need to support overlapping windows; may relocate or hide overlapped ones.
* Assume row/column lock (no font-driven word reflow). Caption providers must set rl=1, cl=1, ww=0.
* Support LEFT/RIGHT/CENTER justification; FULL is optional (treat as LEFT if not implemented).
* Support smooth scroll (in line with CEA-608's 0.20–0.433 s).
* Support SNAP display effect; FADE and WIPE optional.
* Support borderless solid black opaque, and borderless transparent (`fill_opacity=TRANSPARENT, border_type=NONE`). Predefined Window Style #1 + Pen Style #1 are mandatory; #2 is recommended.

### 10.4 Pen (§9.12..§9.18)

* Implement all 3 pen sizes (SMALL, STANDARD, LARGE).
* Implement all 8 font styles (renderer's choice of typeface).
* Support normal, italic, underlined.
* Character offset (sub/superscript) optional.
* Foreground colors: at least 8 (white, black, red, green, blue, yellow, magenta, cyan), with map-down rule.
* Foreground opacity: at least SOLID and FLASH.
* Background: solid black is enough at minimum; should extend the bg box beyond the character to maintain legibility at row edges.
* Edge color and edge type must be controllable independently.

### 10.5 Sync (§9.21)

* Service Input Buffer ≥ 128 bytes.
* On Delay, suspend processing for the specified time; buffer incoming data; resume after timeout, DLC, RST, or buffer overflow.

### 10.6 NTSC output (§9.22, §9.23)

* If the device has an NTSC output, the entire CEA-608 datastream from the DTVCC Transport must be properly re-encoded onto NTSC line 21 per CEA-608.
* Auto-transcoding 708 captions to 608 is not required and not standardized; the embedded 608 datastream is the intended path for 608 viewers.

### 10.7 Use of 608 datastream (§9.24)

* DTV receivers may use 608 as a fallback when 708 is absent.
* If 708 is present, it shall be the default.
* Content advisory data on DTV comes from EIT-0; XDS rating data should not be processed by DTV receivers (but should be passed through to NTSC outputs).

## 11. Encoder behavior

### 11.1 Constant-rate encoding

For systems with fixed 9600 bps DTVCC bandwidth:
* Always include `cc_data()` at every video opportunity. Pad as needed.
* When CEA-608 data is present, every frame's `cc_data()` must contain a CEA-608 byte pair for each field — even when one field is "no waveform" (encoded as `cc_valid=0, cc_type=00 or 01`).
* Compute `cc_count` from frame rate per Table 4 (e.g. 20 at 30i/30p, 10 at 60p, 25 at 24p, 30 at 30i with 3:2 pull-down).
* For pictures with `top_field_first=1`, cc_type=01 precedes cc_type=00; otherwise cc_type=00 precedes 01.

### 11.2 CCP construction

* `sequence_number` increments mod 4 with each CCP.
* `packet_size_code = 0` means a 128-byte CCP; otherwise the CCP is `packet_size_code * 2` bytes total.
* Pack Service Blocks back-to-back in the CCP. Don't cross CCP boundaries within a Service Block.
* End each CCP with a Null Service Block Header if there is room.
* Encoders may shorten CCPs deliberately at splices.

### 11.3 Variable-length C3 commands

* `BOC, COC, EOC` segments must be transmitted in order.
* Maximum payload per segment = 28 bytes (because Service Block holds at most 31 bytes including the EXT1 + opcode + header bytes).

### 11.4 Padding

* Padding byte pairs in `cc_data()` are `cc_valid=0, cc_type=10 or 11`, `cc_data_1 = 0x00, cc_data_2 = 0x00`.
* DTVCC byte stream within CCPs uses NUL (0x00) as the C0 idle filler.

---

## 12. Transport on ATSC (Annex E, informative)

* DTVCC Service Data carried as MPEG-2 Picture User Data (`extension_and_user_data(2)` at the picture level), with `ATSC_identifier = 'GA94'` and `user_data_type_code = 0x03`.
* Caption Service Directory is in `caption_service_descriptor()` in PMT and (per FCC) the EIT.
* Only one `cc_data()`-bearing user_data construct per picture_header.
* `vbv_delay` ≤ 45000 keeps caption latency ≤ 0.5 s.
* SMPTE 334-2 Caption Distribution Packet (CDP) is the file/distribution-format representation of `cc_data()` for ancillary data and for storage.

## 13. Closed-caption data structure overview (Annex G, informative)

The full nesting (top to bottom):

1. `cc_data()` structure: header + N (cc_valid, cc_type, byte_pair) iterations.
2. The leading 1..3 byte_pairs (depending on frame rate) are CEA-608 (cc_type 00/01).
3. The remainder is the DTVCC Caption Channel: a sequence of Caption Channel Packets, each beginning at a `cc_type=11, cc_valid=1` byte pair and continuing with `cc_type=10, cc_valid=1` byte pairs.
4. Each CCP has a 1-byte header (sequence + packet_size_code) followed by `packet_data_size` bytes of CCP data.
5. Within a CCP's data area is a sequence of Service Blocks, each with a 1- or 2-byte header (service_number + block_size [+ extended service number]) followed by `block_size` bytes of block data.
6. Within Service Block data is a stream of byte-codes (graphic chars from G0/G1/G2/G3 and control codes from C0/C1/C2/C3) that form caption commands and text.

## 14. Key gotchas / interpretation notes

* The pop-on caption pattern *does not* use a dedicated caption-flip command (unlike CEA-608's EOC). Instead, you use two windows and TGW.
* `wordwrap` is forced OFF in CEA-708-E. Earlier revisions had it as a real parameter; legacy decoders may behave differently.
* `row_lock` and `column_lock` are forced ON in CEA-708-E. Always YES in encoded streams.
* Extended service numbers start at 7. Service 7 cannot be encoded in a 1-byte (Standard) header — it must use the Extended (2-byte) header form. Standard headers signal services 1–6 only. Service 7 in the standard header (3 bits all 1) is the marker for "extended header follows."
* When parsing, Service Block boundaries do not respect syntactic-element boundaries (a multi-byte command may span Service Blocks within the same CCP). CCP boundaries *do* respect syntactic-element boundaries.
* Service Input Buffer overflow cancels DLY automatically.
* RST clears windows AND the input buffer AND the current window ID.
* Encoded coordinate system (75 × 160/210) ≠ minimum-decoder grid (15 × 32/42); decoders dividing by 5 to map. Caption authors must always use the maximum (5×) coordinate values.
* The opcode bytes 0x93–0x96 are reserved 1-byte commands in C1; decoders must treat them as 1-byte and skip.
* The `cc_count` in `cc_data()` is the number of byte-pair iterations, not bytes.
* The errata to the 2023 publication corrects the SPA example values (pages 77–78 attached at the end of the PDF). The corrected pen-size example for tag-0 dialog is `pen size = 2 = LARGE` and `font style = 2 = Proportional with serifs`. The corrected SPA bit layout is the one in §7.10 above.
