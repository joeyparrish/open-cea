# CEA-608 (CTA-608-E S-2019) — Implementation Digest

This document is a comprehensive digest of ANSI/CTA-608-E S-2019 ("Line 21 Data Services"), sufficient to guide a CEA-608 encoder or decoder implementation without consulting the PDF for technical details. Section numbers in parentheses are cross-references back to the PDF.

The standard is normative for transmissions inserted on line 21 of NTSC video, but the byte-level protocol it defines is the same protocol that is multiplexed inside CEA-708 transport via cc_type = 00 / 01 (see specs/708.md). All content in this digest applies whether the data is being delivered as the analog line-21 waveform, in MPEG-2 user_data, in SEI, in MCC files, or anywhere else.

---

## 1. Scope, terminology, and channel architecture

### 1.1 Services on line 21

Line 21 of the NTSC vertical blanking interval carries one byte pair per field. There are two fields per frame, and the nominal NTSC frame rate is 30/1.001 ≈ 29.97 fps, giving each field a rate of 60/1.001 byte-pairs per second. Each byte is 7 data bits + 1 odd-parity bit (LSB-first NRZ on the wire; in software the parity bit is bit 7 of the octet). Bytes that fail odd parity are treated as invalid and discarded by the decoder.

Within each field, the byte pair may belong to one of several logical *services*. The decoder routes a byte pair to a service based on the field it arrived on and on a small marker (the "data channel" — C1 or C2) embedded in the high bits of certain control bytes. The complete service set is:

| Name  | Field | Channel | Description                                  |
|-------|-------|---------|----------------------------------------------|
| CC1   | F1    | C1      | Primary synchronous caption service          |
| CC2   | F1    | C2      | Special non-synchronous use captions         |
| CC3   | F2    | C1      | Secondary synchronous caption service        |
| CC4   | F2    | C2      | Special non-synchronous use captions         |
| T1    | F1    | C1      | First text service                           |
| T2    | F1    | C2      | Second text service (may carry URLs, §7.10)  |
| T3    | F2    | C1      | Third text service                           |
| T4    | F2    | C2      | Fourth text service                          |
| XDS   | F2    | C3      | eXtended Data Service (F2 only)              |

Captioning has absolute priority over Text and XDS. CC1 is the canonical English/primary-language captioning channel; CC3 is the alternate-language channel. Data must not be duplicated across channels or fields.

### 1.2 Modes

A given service operates in one of three high-level *modes* at any time:

* **Caption Mode** — pop-on, roll-up, or paint-on style captions. Used for CC1–CC4.
* **Text Mode** — full-screen scrolling text in a self-contained box. Used for T1–T4.
* **XDS Mode** — packetized program metadata (F2 only).

Mode is changed by mode-selection control codes (RCL, RDC, RU2/3/4 select Caption Mode and a style; TR and RTD select Text Mode; an XDS class control byte 0x01–0x0E starts an XDS packet). Mode selection is sticky for that service until another mode-selection code is received.

### 1.3 Display geometry

* **Rows 1–15**, top to bottom; row 15 is the bottom row.
* **Columns 1–32**, left to right (columns 0 and 33 exist conceptually for solid-space padding but cannot hold displayable characters).
* Caption captions are always inside the FCC "safe caption area": 80% of picture height (10% top, 10% bottom margin), 106.67% / 133.3% wide (≈80% of picture width with 13.33% / 86.67% horizontal margins).

## 2. Byte format and parity

* Wire format: NRZ, 7 ASCII data bits + 1 odd parity bit. The parity bit is the most significant bit of the octet (bit 7).
* In every reference table in the spec and in this digest, hex codes are shown *without parity* — i.e. the value of the lower 7 bits. Encoders must compute and OR-in the parity bit; decoders must check it and strip it.
* The "odd parity" rule means the total number of '1' bits in the 8-bit octet is odd. For example, the byte 0x00 with odd parity becomes 0x80; the byte 0x14 (5 set bits → already odd) stays 0x14; the byte 0x20 (1 set bit → odd) stays 0x20.
* "Null" character is 0x00; with odd parity it is 0x80. A pair of nulls (0x80, 0x80 on the wire) is the standard idle filler when there is nothing to send.

## 3. Data-channel and field byte mapping

Two independent things are encoded in the high nibble of the first byte of a control pair:

* **Data channel** (channel 1 vs channel 2 within a field) is encoded by adding 8 to the first byte. So channel-1 control codes occupy 0x10..0x17, and channel-2 codes occupy 0x18..0x1F. This holds for *every* category of control code: PACs, Mid-Row codes, Misc Control codes (including Tab Offsets), Special Characters, Extended Characters, and Background/Foreground Attribute codes.
* **Field** (F1 vs F2) is by default *not* encoded in the byte stream — F1 and F2 use the same first-byte values, and the decoder distinguishes them by physical line position (line 21 vs line 284). The lone exception (§8.4) is the Misc Control Codes block: the F1 first bytes 0x14 and 0x1C are remapped to 0x15 and 0x1D on F2. Everything else (PACs, mid-row, special characters, extended characters, attribute codes, Tab Offsets) uses the *same* first-byte values on both fields.

| Category                                 | F1 ch1 first byte(s) | F1 ch2 first byte(s) | F2 ch1 first byte(s) | F2 ch2 first byte(s) |
|------------------------------------------|----------------------|----------------------|----------------------|----------------------|
| Misc Control Codes (0x14/0x1C base)      | 0x14                 | 0x1C                 | 0x15                 | 0x1D                 |
| Misc Control Codes — Tab Offsets         | 0x17                 | 0x1F                 | 0x17                 | 0x1F                 |
| Mid-Row Codes                            | 0x11                 | 0x19                 | 0x11                 | 0x19                 |
| Special Characters (Table 49)            | 0x11                 | 0x19                 | 0x11                 | 0x19                 |
| Extended Characters — Spanish/French (Table 5/7) | 0x12         | 0x1A                 | 0x12                 | 0x1A                 |
| Extended Characters — Portuguese/German/Danish (Tables 8–10) | 0x13 | 0x1B           | 0x13                 | 0x1B                 |
| Background Attribute Codes (Table 3)     | 0x10                 | 0x18                 | 0x10                 | 0x18                 |
| BT / FA / FAU (Table 3)                  | 0x17                 | 0x1F                 | 0x17                 | 0x1F                 |
| PACs (Table 53)                          | 0x10..0x17 per row   | 0x18..0x1F per row   | same as F1 ch1       | same as F1 ch2       |

So a "Resume Caption Loading" (RCL) command is `0x14 0x20` on F1 ch1 (CC1), `0x1C 0x20` on F1 ch2 (CC2), `0x15 0x20` on F2 ch1 (CC3), and `0x1D 0x20` on F2 ch2 (CC4). But a Tab Offset 1 is `0x17 0x21` on every channel of every field — only the line position tells the decoder which of CC1..CC4 it belongs to.

XDS control bytes (0x01..0x0F as a first byte) are F2-only by construction; F1 forbids 0x01..0x0F as a first byte.

In the rest of this document, code pairs are listed in F1 ch1 form unless noted; transpose per the table above for the other channels.

### 4.3 Extended characters — two-byte, with auto-backspace (Tables 5–10)

64 additional characters covering Spanish, French, Portuguese, German, Danish, plus ASCII-compatible characters omitted from the basic set. Prefix bytes (same on both fields — only line position distinguishes F1 from F2):

* Channel 1 (CC1 / CC3): `0x12` for Spanish/French (Tables 5–7) or `0x13` for Portuguese/German/Danish (Tables 8–10).
* Channel 2 (CC2 / CC4): `0x1A` or `0x1B` (the channel-2 +8 offset of 0x12/0x13).

Second byte is `0x20..0x3F`. Each extended character carries an automatic backspace: when received, the cursor moves left one column (overwriting whatever was there) and *then* the extended character is drawn in that cell.

This is important: the conventional encoding pattern is to send a basic-set fallback character first, then immediately send the extended character which backspaces over it and replaces it. For example, to write `ü`, send `'u'` (0x75) followed by the extended-char sequence `0x12 0x25` (F1 ch1). On a decoder that doesn't implement extended chars, the `0x12 0x25` is ignored and the `u` remains. On a decoder that does, the `u` is overwritten by `ü`. So extended chars appear as *two* characters of bandwidth but only one column on screen, and the byte-pair is treated as a *non-printing control pair* by older decoders (does not advance cursor on its own).

The exact characters at each slot:

**0x12/0x1A 0x20..0x27 — Spanish (Table 5):** Á É Ó Ú Ü ü ‘ ¡

**0x12/0x1A 0x28..0x2F — Misc (Table 6):** `*` `'` `—` `©` `℠` `●` `“` `”`

**0x12/0x1A 0x30..0x3F — French (Table 7):** À Â Ç È Ê Ë ë Î Ï ï Ô Ù ù Û « »

**0x13/0x1B 0x20..0x2F — Portuguese (Table 8):** Ã ã Í Ì ì Ò ò Õ õ `{` `}` `\` `^` `_` `|` `~`

**0x13/0x1B 0x30..0x37 — German/misc (Table 9):** Ä ä Ö ö ß ¥ ¤ │ (vertical bar; full-cell-height connector)

**0x13/0x1B 0x38..0x3F — Danish/box-drawing (Table 10):** Å å Ø ø ⌈ ⌉ ⌊ ⌋ (corner pieces; intended to combine with em-dash and vertical bar to draw boxes)

### 4.4 Background and foreground attribute codes (Table 3, §6.2)

These are an *optional* extension defined by EIA-608-D. They appear as a non-printing pair preceded by a standard space; on a basic decoder the space is shown and the attribute code is ignored, on an extended decoder the attribute code's auto-backspace replaces the just-displayed space with a *space colored according to the attribute*. Use is the same idiom as extended characters: send `' '` (0x20), then the attribute code; an older decoder shows the space and skips the code, a newer decoder backspaces and shows a colored space.

Background attribute codes (F1 ch1 first byte = 0x10; ch2 = 0x18):

| 0x24/5   | BBO/BBS  | Background Blue                          |
| 0x26/7   | BCO/BCS  | Background Cyan                          |
| 0x28/9   | BRO/BRS  | Background Red                           |
| 0x2A/B   | BYO/BYS  | Background Yellow                        |
| 0x2C/D   | BMO/BMS  | Background Magenta                       |

The "closed-group extension" assignments in Table 4 (0x17/0x1F + 0x24..0x2A) select alternate two-byte character sets (CJK, etc.); these are not compatible with the FCC-mandated decoder profile and should not appear in North American distribution.

## 5. Caption-mode control codes

All caption-mode and text-mode control pairs are two bytes whose first byte is in `0x10..0x1F` — channel-1 codes in 0x10..0x17, channel-2 codes in 0x18..0x1F. F2 uses the same first-byte values as F1 except for the misc-control 0x14/0x1C → 0x15/0x1D remapping noted in §3. Per FCC rules and the recommendation of §D.2, *each* such control pair must be transmitted twice in successive frames. Decoders act on the *first* of the redundant pair; the second is suppressed silently. This redundancy protects against single-frame data dropouts.

### 5.1 Preamble Address Codes (PACs) — Table 53

A PAC simultaneously sets the cursor row, the indent on that row (0/4/8/12/16/20/24/28), and the initial pen attributes (color or italic, underline-on or underline-off). It does *not* directly draw or erase.

The first byte selects the row; the second byte selects color+underline+indent. The row encoding is irregular (it is not row-number-in-binary) — read row from the first-byte / second-byte combination using the table below. Only F1-channel-1 first bytes are listed; convert per §3.

| 2   | 0x11                | 0x40..0x5F or 0x60..0x7F (same first byte; 2nd-byte high bit selects 1 vs 2) |
| 3   | 0x12                | 0x40..0x5F or 0x60..0x7F |
| 4   | 0x12                | as above           |
| 5   | 0x15                | as above           |
| 6   | 0x15                | as above           |
| 7   | 0x16                | as above           |

| 10  | 0x17                | as above           |
| 11  | 0x10                | 0x40..0x5F or 0x60..0x7F (rows 1–4 + 12–15 use 0x11/0x12/0x14; row 11 uses 0x10) |

Indenting PACs (0x50..0x5F and 0x70..0x7F second-byte values) implicitly set color = WHITE, italic = OFF, and the underline bit per the LSB. PACs are non-destructive in pop-on and paint-on (they relocate cursor but do not erase). In roll-up, a PAC whose row differs from the current base row *immediately* relocates the entire roll-up window to the new base row.

PACs interact with later displayable characters: a displayable character placed in a cell whose immediate left neighbor is a non-empty character inherits the *left neighbor's* attributes, regardless of the PAC. If a character lands on a cell that already holds a PAC or mid-row code, characters already to the right of that cell inherit the new character's attributes (which may visibly recolor a whole row).

### 5.2 Mid-Row codes (Table 51)

A mid-row code is a non-printing pair that sets the pen color/italic and underline state from the *next* column onward. Unlike a PAC, a mid-row code occupies one column on screen — it appears as a space whose effect is to apply the new attributes to subsequent characters. Mid-row codes turn off italics for color codes and turn off italics-underline for non-italic codes.

* First byte F1 ch1 = `0x11`; F1 ch2 = `0x19`.
* Second byte 0x20..0x2F:

| 2nd byte | Description       |
|----------|-------------------|
| 0x20     | White             |
| 0x21     | White Underline   |
| 0x22/3   | Green / Underline |
| 0x24/5   | Blue / Underline  |
| 0x26/7   | Cyan / Underline  |
| 0x28/9   | Red / Underline   |
| 0x2A/B   | Yellow / Underline|
| 0x2C/D   | Magenta / Underline|
| 0x2E     | Italics (white)   |
| 0x2F     | Italics Underline |

(Note: there is no italic-color combination — italic is forced to white.)

### 5.4 Roll-up base row, EDM/ENM, EOC semantics

* RU2/3/4 sets roll-up style and depth, and moves the cursor to column 1 of the current base row. It also *erases* any previously displayed pop-on or paint-on caption (FCC-mandated).
* In roll-up, the *base row* is the bottom row of the rolling window. The base row defaults to row 15 if no PAC has been seen; otherwise it is whatever row a PAC has set. The window contains `depth` rows above the base row. New text always lands on the base row; CR scrolls everything up by one row.
* EDM erases the currently displayed caption (any style). ENM erases the non-displayed memory. Both can stand alone, no preceding RCL/RDC needed; they apply to caption memory regardless of which mode is active (even during Text Mode).
* EOC swaps displayed and non-displayed memory; this is the trigger that makes a pop-on caption appear. EOC also forces the service into pop-on style (because EOC is meaningless otherwise). EOC during a roll-up display is permitted by the most recent FCC orders: it swaps the roll-up caption into non-displayed memory but does not rearrange rows. Service providers should not normally do this.
* TR (Text Restart) clears all text on the text screen and moves the cursor to row 1 column 1.

### 5.5 Style-switching matrix

* RCL — selects pop-on. No memory side effects. Following data lands in non-displayed memory.
* RDC — selects paint-on. No memory side effects. Following data is painted directly to the displayed memory.
* RU2/RU3/RU4 — selects roll-up with given depth. **Erases both displayed and non-displayed memory** if the previous style was pop-on or paint-on. Cursor goes to col 1 of base row.
* EOC — flip displayed/non-displayed; forces pop-on. Does not erase.
* TR — clear text screen, switch to Text Mode.
* RTD — switch to Text Mode without clear.

### 5.6 Backspace, column 32, BS edge case

* BS at column 1 is a no-op.
* If a character is in column 32 (the rightmost), BS moves cursor to column 31 and erases that cell. Decoders may *additionally* erase column 32 (TC2 does this); both behaviors are legal. To portably erase a character in column 32, encoders should send a transparent space or DER instead of relying on BS semantics in column 32.
* Once the cursor reaches column 32, additional incoming characters overwrite column 32 (they do not advance the cursor past 32).

### 5.7 Carriage Return semantics by mode

* **Roll-up:** scroll the displayed window up by one row, erase the (new) base row, place cursor at column 1 of the base row.
* **Paint-on:** no scroll; cursor moves but the spec does not require display scroll. In practice, decoders move the cursor to column 1 of the next row (downward).
* **Pop-on:** CR is generally not used; some implementations move the cursor.
* **Text mode:** scroll the text window up by one row, place cursor at column 1.

The "cursor at column 1" default is overridden by any subsequent PAC.

### 5.8 Smooth scroll

The spec allows but does not require smooth scroll. If implemented, a roll-up scroll must take between 0.20 s and 0.433 s (6–13 frames), in 6–13 discrete steps. If a CR arrives mid-scroll, the scroll is terminated and rows snap to final positions. Characters received during the scroll should be drawn incrementally, not buffered until the scroll finishes.

## 6. Caption-mode display behaviors

### 6.1 Memory model

The decoder maintains two caption memories per service: *displayed* and *non-displayed*. Pop-on captions are loaded into non-displayed memory and made visible by EOC (which swaps the two memories). Roll-up and paint-on write directly to displayed memory.

A row is considered "allocated" when a *displayable character* (or solid-space attribute code) lands on it; not when a PAC is received. The FCC mandates that decoders display at least 4 rows simultaneously. Once 4 rows are allocated, the only way to free a row of memory is to position the cursor at column 1 of that row and send DER. Other erasure techniques (BS, mid-row DER) may also free a row at the manufacturer's option but are not guaranteed to.

### 6.2 Attribute inheritance

When a displayable character is written into a cell:
1. If the cell immediately to the left contains a displayable character (or mid-row code with attributes), the new character inherits those attributes — even if the new character was just preceded by a PAC. The PAC's attributes are ignored for that character.
2. Otherwise the new character takes the attributes specified by the most recent PAC.
3. If a displayable character overwrites an existing PAC or mid-row code with characters already to its right, those right-hand characters take on the new character's attributes (visible recolor of trailing text).
4. After a DER, if a character is then written at the cursor and there is a character to its left, the new character takes the deleted-region's prior attributes; otherwise the most recent PAC's attributes apply.
5. Default attributes when no PAC has been received: white, non-underlined, non-italic, non-flash, opaque-black background.

### 6.3 Solid spaces (legibility padding)

A decoder may add one solid-color space cell at the leftmost (column 0 conceptually) and rightmost (column 33 conceptually) edges of each caption row, to give the text a visible margin. This is optional; if implemented, both leading and trailing must be done for symmetry. Mid-row codes and FON, since they take a column on screen, also count as displayable cells for this purpose.

### 6.4 Display enable/disable timing

* Enable: the decoder must wait until *n* consecutive frames of valid (odd-parity) data are received, where 12 ≤ n ≤ 18 (0.40 s to 0.60 s). A frame with even parity or no data resets the count. The data accumulated during the n frames is preserved and displayed as soon as enable is reached.
* Disable: 45 consecutive frames of no-data or even-parity bytes triggers disable; a frame of valid data resets the count.

### 6.5 Automatic caption erasure

Optional: if no caption data arrives for ≥ 16 seconds in a service, the decoder may automatically erase the displayed memory. This must not be applied to non-displayed memory (pop-on captions may legitimately be pre-loaded for up to 15 s before EOC). Text Mode is not subject to this auto-erasure.

### 6.6 Color in monochrome decoders

Monochrome decoders display all characters in a single color. They must still honor the *spacing/positioning* effects of mid-row and PAC codes (the column they take, the underline bit, italic-attribute width effects), but ignore the color choice.

### 6.7 Prompt response

Decoders must respond to a valid control pair within one video frame of receipt of the first of the redundant pair.

## 7. Text Mode (§7)

Text Mode displays a continuous stream of text inside a fixed window. The window is at least 7 and at most 15 rows high, at least 32 columns wide, with a *required* solid background (regardless of the per-character background attribute). Text Mode is initiated by RTD (resume) or TR (restart, with clear); the initial cursor is at the topmost row, column 1.

Once data fills the window, additional CRs scroll the window up exactly like roll-up captions (smooth-scroll within 0.433 s). The cursor cannot be moved arbitrarily within Text Mode: it auto-advances after each character or mid-row code; PACs only set the *indent* (the row number is ignored in Text Mode); BS moves the cursor left one column and erases that cell; DER erases from the cursor to end of row; TR clears the entire screen.

Text Mode is terminated only by these caption-specific control codes in the same data channel: EOC, RCL, RDC, RU2, RU3, RU4. EDM and ENM, FON, BS and unknown PACs/mid-row codes do *not* terminate Text Mode. On F2, a Text Mode stream is also terminated by an XDS control byte (0x01..0x0F as a first byte).

There is no automatic memory erasure in Text Mode (in contrast to Caption Mode's optional 16-second timeout).

### 7.1 URL transport in T-2 (§7.10–§7.14)

T-2 (T2) may carry URLs. The character set inside a URL is interpreted as ISO-8859-1, *not* CEA-608 — bytes 0x20..0x7E mean what ASCII says, not what 608's basic-character table says. Bytes outside 0x20..0x7E are discarded; second bytes of two-byte 608 control codes are also discarded.

A URL is wrapped in angle brackets and may be followed by attribute/value pairs in square brackets and a 16-bit checksum:

```
<url>[attr1:val1][attr2:val2]...[attrn:valn][checksum]
```

Attributes:

* `type:` (or `t:`) — `program | network | station | sponsor | operator` (single-char abbreviations: `p / n / s / a / o`)
* `name:` (or `n:`) — human-readable title; cannot contain `[` `]` `<` `>`
* `expires:` (or `e:`) — ISO-8601 date/time, e.g. `19991231T115959`
* `script:` (or `s:`) — ECMAScript fragment

Checksum: pair adjacent bytes (starting at the leading `<`) into 16-bit big-endian integers, padding with a zero byte if the count is odd. The checksum is the value such that the one's-complement sum of all those 16-bit integers plus the checksum equals 0xFFFF (the IETF Internet checksum, RFC 1071). It is encoded as four hex digits in square brackets at the end. The square brackets and the checksum hex digits themselves are not part of the calculation. Bytes outside 0x20..0x7E are excluded from the checksum.

Special characters not in ISO-8859-1 0x20..0x7E or excluded by the syntax must be percent-encoded (`%XX`).

URL transmissions on T-2 should not consume more than 25% of F1 bandwidth so as to leave room for downstream services.

If a URL is interrupted by CC1/CC2/T1, the encoder must resume by sending RTD (not TR) before continuing the URL. A receiver may treat a TR while expecting a URL continuation as terminating the URL.

### 7.2 Future enhanced-Text identifiers (Annex G.2)

Article identifiers (informational expansion):

* T1/T3: `0x16 0x20` = ANS (Article Name Start), `0x16 0x30` = ANE (Article Name End), `0x16 0x31` = AC (Article Clear), `0x16 0x32` = AE (Article End).
* T2/T4: same suffixes with first byte `0x1E`.
* `0x16 0x21..0x2F` (and `0x1E 0x21..0x2F`) are page identifiers for pages 2–16 within an article.

These are forward-compatibility codes; legacy decoders ignore them.

## 8. Field-2 / XDS (§8 and §9)

XDS is an F2-only metadata channel. It coexists with F2 captioning and F2 text and is *interleaved* on a space-available basis. Captioning has top priority on F2; XDS and text share the remainder roughly equally.

### 8.1 XDS framing

* All XDS bytes must have odd parity.
* "Control" bytes are 0x01..0x0F as the *first* byte of a pair. They mark the boundaries of XDS sub-packets:

| Byte | Function | Class                |
|------|----------|----------------------|
| 0x01 | Start    | Current              |
| 0x02 | Continue | Current              |
| 0x03 | Start    | Future               |
| 0x04 | Continue | Future               |
| 0x05 | Start    | Channel              |
| 0x06 | Continue | Channel              |
| 0x07 | Start    | Miscellaneous        |
| 0x08 | Continue | Miscellaneous        |
| 0x09 | Start    | Public Service       |
| 0x0A | Continue | Public Service       |
| 0x0B | Start    | Reserved             |
| 0x0C | Continue | Reserved             |
| 0x0D | Start    | Private Data         |
| 0x0E | Continue | Private Data         |
| 0x0F | End      | (any class)          |

* The byte that follows a Start or Continue control byte is the *Type* byte (range 0x01..0x7F). For a given (Class, Type) the Type byte must match between Start and any subsequent Continue bytes for the same packet.
* Bit 6 of the Type byte distinguishes In-Band (0) vs Out-of-Band (1).
* Informational characters fill bytes after the Type byte. They are limited to 0x00 or 0x20..0x7F for displayable text; for binary fields, b6 must be set (i.e. value must be in 0x40..0x7F or otherwise have b6 set). Encodings 0x01..0x1F as informational bytes are forbidden.
* A packet ends with `0x0F` (End control) followed by a single Checksum byte. The End/Checksum pair is the only one that terminates the packet — there is exactly one End code regardless of class.
* Maximum 32 informational bytes per packet. The total informational length must be even; pad with 0x00 if needed.
* XDS control codes are *not* doubled (in contrast to caption-mode control codes).
* When an XDS packet is interleaved with caption data, the caption data resumes by the encoder sending the appropriate caption-mode control pair (RCL/RU2/RU3/RU4/RDC/RTD). The XDS packet may be suspended and continued via Continue.
* Up to one level of nested interleaving (XDS-inside-XDS) is permitted; deeper nesting is not recommended.

Checksum: the two's-complement value such that the 7-bit sum (mod 128) of (Start byte) + (Type byte) + all informational bytes + End byte + Checksum equals 0. Continue/Type bytes are *not* counted. So if S = sum of (Start, Type, all info bytes, End), then Checksum = (-S) mod 128.

### 8.2 Class / Type catalog (§9.5)

Only the most common types are summarized here; less common types are listed by Class+Type with a one-line description.

#### 8.2.1 Current Class (0x01 Start, 0x02 Continue) — info about the program currently airing.

* **0x01 Program Identification Number ("scheduled start time")** — 4 bytes encoding minute/hour/date/month with bit 6 set in each byte. Layouts of bits 0..5 (low-to-high):
  - Minute byte: bits 0..5 = minute (0..59); bit 5..0 → minute. Bit 6 = 1.
  - Hour byte: bits 0..4 = hour (0..23); bit 5 = D (DST flag, ignored in this packet, used by Time of Day). Bit 6 = 1.
  - Date byte: bits 0..4 = date (1..31); bit 5 = L (used by Time of Day for Feb 28/29). Bit 6 = 1.
  - Month byte: bits 0..3 = month (1..12); bit 4 = T (tape-delay flag); bit 5 = Z (Time of Day "zero seconds" flag). Bit 6 = 1.
  - All 4 bytes set to all-ones (0x7F each, i.e. 0xFF after parity) signals end of current program.
  - A change in the received Program ID for the Current class is interpreted as the start of a new program; receivers should discard prior current-class info.
* **0x02 Length / Time-in-Show** — 2, 4, or 6 binary bytes: length(min, hour); elapsed(min, hour); elapsed(sec, null). All have b6 = 1 except the trailing standard-null. Range: minute and second 0..59, hour 0..23.
* **0x03 Program Name (Title)** — 2..32 displayable bytes (0x20..0x7F). A change is interpreted as a new program.
* **0x04 Program Type** — 2..32 keyword bytes from Table 17. First seven values (0x20..0x26) are the "Basic" group (Education, Entertainment, Movie, News, Religious, Sports, OTHER). Codes 0x27..0x7F are the "Detail" group, listed alphabetically in the spec (Action, Advertisement, Animated, Anthology, Automobile, Awards, Baseball, ... Western). Basic codes must precede Detail codes. Five or fewer total codes recommended for Composite-1 efficiency.
* **0x05 Content Advisory** — 2 bytes encoding rating system + age + content-flag bits per Table 18:
  - Char1: bit6=1, bit5 = D/a2, bits4..3 = a1/a0, bits 2..0 = r2/r1/r0.
  - Char2: bit6=1, bit5 = (F)V, bit4 = S, bit3 = L/a3, bits 2..0 = g2/g1/g0.
  - System-selector bits a3..a0 select among MPA, U.S. TV Parental, Canadian English, Canadian French, and reserved systems (Table 19). Tables 20–23 give the per-system age and content flag encodings, including invalid combinations that decoders must treat as invalid packets (e.g. Canadian English (g2,g1,g0)=(1,1,1) and Canadian French (g2,g1,g0)=(1,1,0) and (1,1,1)).
  - Within the U.S. TV system, the V bit is repurposed as FV when age=TV-Y7.
  - Content advisory data must remain unchanged for the duration of a program (including commercials/promos within); it is cleared only when the Program ID or Program Name changes.
* **0x06 Audio Services** — 2 bytes (Main, SAP). Each: b6=1, bits 5..3 = language code (Table 25: Unknown, English, Spanish, French, German, Italian, Other, None), bits 2..0 = type code. Main types: Unknown, Mono, Simulated Stereo, True Stereo, Stereo Surround, Data Service, Other, None. SAP types: Unknown, Mono, Video Descriptions, Non-program Audio, Special Effects, Data Service, Other, None.
* **0x07 Caption Services** — 2..8 bytes; one byte per available service. Each byte: b6=1, bits 5..3 = language, bit 2 = F (field 2 vs 1), bit 1 = C (channel 2 vs 1), bit 0 = T (text vs caption). 8 combinations cover CC1/CC3/CC2/CC4/T1/T3/T2/T4.
* **0x08 Copy and Redistribution Control** — 2 bytes. Byte1 (b6=1): - reserved, CGMS-A bits b4..b3, APS bits b2..b1, ASB bit b0. Byte2 (b6=1): all reserved except b0 = RCD (redistribution control descriptor flag from ATSC A/65). CGMS-A meanings: 00=copy free; 01=no more copies; 10=one generation; 11=no copy. APS meanings: 00=no APS; 01=PSP/no-split-burst; 10=PSP+2-line split burst; 11=PSP+4-line split burst.
* **0x09** Reserved.
* **0x0C Composite Packet-1** — efficient bundle: Program Type (5), Content Advisory byte 1 only (1), Length (2), Time-in-Show (2), Title (0..22). Total ≤ 32, even. (Title >22 chars must use the separate Title packet 0x03.)
* **0x0D Composite Packet-2** — Program Start Time ID (4), Audio Services (2), Caption Services (2), Call Letters (4), Native Channel (2), Network Name (0..18). Total ≤ 32, even. (Network name >18 must use separate Network Name packet.)
* **0x10..0x17 Program Description Row 1..Row 8** — each is 0..32 bytes of free-form text describing the current program (episode title, synopsis, cast, etc.). A change in any row is interpreted as a new program.

#### 8.2.2 Future Class (0x03 Start, 0x04 Continue)

Same packet types as Current Class. The first packet of a Future-class data set must be the Program ID (0x01) with the start time of that future program; that ID is the index for all subsequent Future-class data about that program.

#### 8.2.3 Channel Information Class (0x05 Start, 0x06 Continue) — info about the channel itself

* **0x01 Network Name** — 2..32 displayable bytes.
* **0x02 Call Letters / Native Channel** — 4 bytes (call letters; pad with 0x20 if 3-letter call sign) optionally followed by 2 bytes (native channel number, in displayable digits).
* **0x03 Tape Delay** — 2 binary bytes (minutes, hours; same format as Program ID minute/hour). Default 0 if absent. Applies to programs whose ID has T=1.
* **0x04 TSID (Transmission Signal Identifier)** — 4 binary bytes encoding 16-bit TSID nibble-by-nibble. Bits 5..4 of each byte reserved.

#### 8.2.4 Miscellaneous Class (0x07 Start, 0x08 Continue)

* **0x01 Time of Day** — 6 binary bytes: minute, hour (with D=DST observed flag), date (with L = Feb 28/29 disambiguation), month (with Z = "zero the seconds counter"), day (1..7, 1=Sunday), year (offset from 1990). Always UTC. Z bit may be set on at most one Time of Day per minute.
* **0x02 Impulse Capture ID** — 6 bytes: Program ID (4 bytes) + first 2 bytes of Length/Time-in-Show. Used to flag a future recording trigger.
* **0x03 Supplemental Data Location** — 2..32 bytes; each byte (b6=1) has bit5 = field flag (1=F2, 0=F1) and bits4..0 = VBI line number (7..31).
* **0x04 Local Time Zone & DST** — 2 bytes. Byte1 b6=1, b5=D (DST observed by entire viewing audience), bits4..0 = hour offset westward from UTC (e.g. EST=5). Byte2 = standard null. Must not be transmitted if signal can be received in multiple time zones.
* **0x40 Out-of-Band Channel Number**, **0x41 Channel Map Pointer**, **0x42 Channel Map Header**, **0x43 Channel Map Packet** — cable channel mapping subsystem (informational; recommendations in Annex M).

#### 8.2.5 Public Service Class (0x09 Start, 0x0A Continue)

* **0x01 NWS WRSAME Code** — emergency-broadcast packet using NWS Specific Area Message Encoder format. Body: "EEE-PSSCCC-+TT-" then end. EEE = 3-letter event code (Table 44 lists ~50: TOR Tornado Warning, FFW Flash Flood Warning, EAN Emergency Action Notification, etc.). P = subcounty digit 1..9. SS = state FIPS code. CCC = county FIPS code. TT = valid period in 15-minute units. NWS Code packets must be transmitted at high priority for the duration of the message validity period and must not be deleted by downstream encoders.
* **0x02 NWS Message** — free text (multi-packet, 32-byte chunks).

#### 8.2.6 Reserved Class (0x0B/0x0C) and Private Data Class (0x0D/0x0E)

Reserved is undefined. Private Data is for closed systems and not standardized.

### 8.3 XDS encoder / decoder behavior

* Updated info simply replaces older info; there is no "delete" mechanism. Receipt of a new Program ID for the Current class clears all current-class state for the receiver.
* Repetition rates (Annex E.7.3, advisory):
  - High priority (every 2–5 s if data is present): Composite 1, Composite 2, Length/TIS, Title, Aspect Ratio, Content Advisory; Network Name, Call Letters/Native, TSID; Impulse Capture; NWS Code; CGMS-A. Content Advisory ≤ 3 s; CGMS-A ≤ 5 s.
  - Medium (10–30 s): Program ID, Program Type, Audio Services, Caption Services, Program Descriptions 1..8; Channel Map Pointer/Header/Map.
  - Low (1–10 min): all Future class; Tape Delay; Time of Day; Supplemental Data Location; Time Zone; Out-of-Band Channel; NWS Message.
* Priority order for inserting into F2 bandwidth: F2 captions first, then Time-of-Day with Z-bit set, then Content Advisory if it has been ≥3 s, then CGMS-A if it has been ≥5 s, then NWS Code, then anything else.
* Encoders should prefer Composite-1/Composite-2 packets over their constituent single-purpose packets when the data fits (saves bandwidth). Exception: Content Advisory's full second byte still needs the standalone packet (only the first byte fits in Composite-1).
* Time of Day must be transmitted as close to top-of-minute as possible.
* Buffer expiration timers (Table 47) for each XDS type: NWS Code = NWS valid period; NWS Message = transmit once; Impulse Capture = length of promo; Current Class = 5 min capped to time remaining (resets when Program ID changes); Future Class = until program start; Channel Info, Time of Day, etc. = forever / operator-set; undefined classes = retransmit once within 10 s of receipt.
* Captions take *immediate* priority on F2 — any in-progress XDS or text packet must be suspended at the next byte-pair boundary and resumed afterward.

## 9. Encoder requirements

### 9.1 Universal

* All caption-mode and text-mode control pairs (i.e. those whose first byte is in 0x10..0x1F or 0x18..0x1F) must be transmitted twice in successive frames. Insert null bytes (0x00 with parity = 0x80) as needed to keep two-byte control pairs frame-aligned (both bytes of a pair must be in the same field, and a control pair must start at frame even-pair boundary).
* All bytes must have odd parity forced.
* Field 2 should NOT use redundant doubling for XDS data; only for caption/text control codes.
* Control codes 0x01..0x0F are *forbidden* on F1 as the first byte of a pair (those slots are reserved for F2 XDS).
* Encoders must not assume larger than 4-byte (two byte-pair) per-field decoder buffers when scheduling.
* Each encoder must keep CC1/CC2 captioning delay relative to video at ≤ 2 frames; CC3/CC4 at ≤ 4 frames.
* Field 2 text bandwidth: limit packets to 40 bytes; estimate that captions take 40% of bandwidth, text and XDS share 30%/30%; insert delay between text packets equal to `packet_size / available_text_bandwidth_fraction`, rounded up to next even number (e.g. 26-byte packet at 30% text bandwidth → 88 byte delay).
* If F2 is used, F1 must contain at least valid clock-run-in + null bytes, even when F1 has no caption/text data.
* Encoder must preserve and pass through upstream XDS data unmodified unless explicitly instructed otherwise.

### 9.2 Pop-on encoding pattern

```
RCL                  (select pop-on)
ENM                  (optional: clear non-displayed memory)
PAC (row,indent,attr)
text...
PAC                  (next element)
text...
EDM                  (optional: clear current display before flip)
EOC                  (display the new caption)
```

EDM-before-EOC introduces a brief blink between captions (often desirable). ENM-after-EOC is the alternative; only EDM-before is supported by TC1, but TC1 is no longer relevant.

The first EOC in a redundant pair is the visible "appear time" — encoders should align that EOC to the desired display frame.

### 9.3 Roll-up encoding pattern

For each new line of text:
```
RU2 / RU3 / RU4      (set roll-up + depth; only effective on first line)
CR                   (scroll up one row)
PAC (base_row, indent, attr)
text...
```

To relocate the rolling window mid-stream, send a new PAC whose row differs from the current base row.

### 9.4 Paint-on encoding pattern

```
RDC                  (select paint-on)
PAC (row, indent, attr)
text...
PAC                  (relocate cursor; or DER to delete row)
text...
```

If transmission has been paused, restart each burst with RDC.

### 9.5 Line-21 waveform (informational here)

For software encoders writing the analog waveform:
* Clock run-in is 7 cycles of a 0.5034965 MHz sine wave centered on 25 IRE.
* 3 start bits "0,0,1" precede the data bits.
* Each bit is a 1.986 µs cell. Data bit logic-0 is 0 IRE, logic-1 is 50 IRE.
* H-Sync to clock-run-in is nominally 10.500 µs. Two characters (16 data bits) follow.
* Field 2 = line 284 in NTSC numbering.

For digital embedding, the byte-level protocol (everything in §1–§8 of this digest) is what matters.

## 10. Decoder requirements (Annex C)

* Display ≥ 4 caption rows simultaneously, anywhere on screen (FCC).
* Implement the BS-in-column-32 behavior (cursor → col 31, erase col 31; col 32 may also be erased).
* Smooth-scroll roll-up captions in 0.20–0.433 s if smooth scroll is supported.
* Resolve roll-up depth vs base-row conflicts in favor of the depth (don't lose data).
* Allocate row memory upon first displayable character (not on PAC arrival).
* Free row memory only on DER at column 1 (other erasure techniques may free memory at the manufacturer's option).
* If a 5th row of caption data is received without intervening erase/DER, display as much as possible; do not drop newer rows. One acceptable solution: erase displayed memory and use the 5th row as the start of a new caption.
* Respond to commands within 1 frame.
* Style switching: RCL/RDC/EOC must not erase memory by themselves; RU2/3/4 must erase pop-on/paint-on memory if changing style. EDM erases displayed; ENM erases non-displayed; both work regardless of mode.
* Cursor location is per-mode and per-memory: the decoder remembers where the cursor was for caption-mode vs text-mode separately, but only one cursor for displayed-vs-non-displayed caption memory.
* For windowless roll-up after CR with no PAC: cursor goes to column 1 of the same base row (or row 15 if no PAC has been seen). Receipt of a Roll-Up command followed by a character with no intervening PAC/CR while a roll-up is already displayed should NOT reset the base row.
* Display enable: 12–18 frames consecutive valid. Disable: 45 frames consecutive invalid/no-data.
* In TV-receiver implementations, content-advisory blocking (per FCC §15.120) must mute audio, blank video, and eliminate program-related captions when triggered.

## 11. Compliance / errata gotchas

* Annex H, B.9, B.10, B.11.1 (TeleCaption I/II support schedule) are historical; ignore the "shalls" there. Only support the FCC-rules feature profile.
* Annex J (repetition-rate tables) is informative; the values in §E.7.3 are the authoritative recommended rates.
* Annex F.2's text quoting `EIA-608` and `EIA-744` references is from older FCC orders; the substantive blocking requirements still apply.
* Code-pair doubling is mandatory on F1 for control pairs; on F2 for caption/text control pairs only; XDS control pairs are NOT doubled.
* Background attribute codes (Table 3) and the auto-backspace mechanism are *not* part of the FCC mandate; they were added by EIA-608-D as an option.
* The 0x7E Annex A entry says "Ñ" but Table 50 (FCC mandated) places `ñ` at 0x7E and `Ñ` at 0x7D; trust Table 50.
